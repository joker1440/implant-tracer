# Implant Case Manager - Supabase Schema Proposal

## Design principles

- Single-user app, but keep `owner_user_id` on core tables so RLS is clean and future expansion is still possible.
- Model `Patient -> Case -> Visit` explicitly.
- A case is always tied to exactly one tooth position.
- A visit can contain one or more procedures, because implant cases often combine acts on the same day.
- Planned follow-up steps are stored separately from completed visits, so overdue/upcoming dashboards stay simple.
- Photos live in Supabase Storage, with metadata stored in Postgres.

## Core relationships

```text
auth.users
  └─ patients
       └─ cases
            ├─ case_plan_steps
            ├─ visits
            │    ├─ visit_procedures
            │    └─ visit_photos
            └─ case_implants

tooth_positions
treatment_templates
  └─ treatment_template_steps
```

## Tables

### `patients`

One patient can have many implant cases.

Suggested columns:

- `id uuid primary key default gen_random_uuid()`
- `owner_user_id uuid not null references auth.users(id)`
- `full_name text not null`
- `phone text`
- `birth_date date`
- `chart_no text`
- `attention_alert text`
- `general_notes text`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Notes:

- `attention_alert` is the global warning field shown everywhere in the UI.
- Add a trigram index on `full_name` for fast search.

### `tooth_positions`

Seed table for the tooth picker.

Suggested columns:

- `code text primary key`
- `label text not null`
- `arch text not null check (arch in ('upper', 'lower'))`
- `quadrant smallint not null`
- `sort_order smallint not null`

Seed with FDI permanent teeth:

- Upper right: `18` to `11`
- Upper left: `21` to `28`
- Lower left: `38` to `31`
- Lower right: `41` to `48`

Assumption:

- Use FDI numbering, which is common for dental charting here. If you want Universal numbering instead, we should switch before implementation.

### `treatment_templates`

Seed table for selectable treatment flows.

Suggested columns:

- `key treatment_template_key primary key`
- `label text not null`
- `description text`
- `is_active boolean not null default true`

Example rows:

- `arp_to_implant` / `ARP -> Implant`
- `gbr_to_implant` / `GBR -> Implant`
- `iip` / `IIP`
- `healed_ridge` / `Healed Ridge`
- `sinus_lift_to_implant` / `Sinus Lift -> Implant`

### `treatment_template_steps`

Default steps used to auto-generate the planned schedule when a case is created.

Suggested columns:

- `id uuid primary key default gen_random_uuid()`
- `template_key treatment_template_key not null references treatment_templates(key) on delete cascade`
- `step_order integer not null`
- `title text not null`
- `procedure_type procedure_type not null`
- `default_offset_days integer`
- `default_note text`
- `created_at timestamptz not null default now()`

Notes:

- `default_offset_days` is the suggested gap from the previous step or case start.
- The UI should still allow editing planned dates after generation.

### `cases`

Each implant case belongs to one patient and one tooth position.

Suggested columns:

- `id uuid primary key default gen_random_uuid()`
- `case_no bigint generated by default as identity unique`
- `owner_user_id uuid not null references auth.users(id)`
- `patient_id uuid not null references patients(id) on delete cascade`
- `tooth_code text not null references tooth_positions(code)`
- `status case_status not null default 'active'`
- `template_key treatment_template_key references treatment_templates(key)`
- `title text`
- `started_on date`
- `target_restoration_on date`
- `diagnosis_notes text`
- `internal_notes text`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Notes:

- `case_no` gives you the easy `#21` style display number in the UI.
- Do not enforce one active case per tooth yet; retreat cases are possible.

### `case_plan_steps`

Stores planned next visits and drives overdue / upcoming dashboards.

Suggested columns:

- `id uuid primary key default gen_random_uuid()`
- `owner_user_id uuid not null references auth.users(id)`
- `case_id uuid not null references cases(id) on delete cascade`
- `template_step_id uuid references treatment_template_steps(id)`
- `step_order integer not null`
- `title text not null`
- `procedure_type procedure_type not null`
- `planned_date date`
- `status plan_step_status not null default 'pending'`
- `note text`
- `completed_visit_id uuid`
- `completed_at timestamptz`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Notes:

- Overdue is calculated, not stored: `status = 'pending' and planned_date < current_date`.
- When the user clicks "complete this step", create a visit and link it back through `completed_visit_id`.

### `visits`

Completed follow-up records.

Suggested columns:

- `id uuid primary key default gen_random_uuid()`
- `owner_user_id uuid not null references auth.users(id)`
- `case_id uuid not null references cases(id) on delete cascade`
- `plan_step_id uuid references case_plan_steps(id) on delete set null`
- `visited_on date not null`
- `summary text`
- `next_note text`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Notes:

- `summary` is the main visit memo.
- `next_note` is optional free text if a special next-step reminder is needed beyond structured plan steps.

### `visit_procedures`

One visit may contain multiple procedures. This is where procedure-specific fields live.

Suggested columns:

- `id uuid primary key default gen_random_uuid()`
- `visit_id uuid not null references visits(id) on delete cascade`
- `procedure_order integer not null default 1`
- `procedure_type procedure_type not null`
- `procedure_note text`
- `implant_brand text`
- `implant_model text`
- `implant_diameter_mm numeric(4,1)`
- `implant_length_mm numeric(4,1)`
- `bone_graft_materials text[] not null default '{}'`
- `membrane_type membrane_type`
- `membrane_note text`
- `sinus_lift_approach sinus_lift_approach`
- `extra_data jsonb not null default '{}'::jsonb`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Notes:

- `implant_*` fields are used only when `procedure_type = 'implant_placement'`.
- `bone_graft_materials`, `membrane_type`, and `sinus_lift_approach` are used for `arp`, `gbr`, and `sinus_lift`.
- `extra_data` keeps room for later procedure-specific fields without schema breakage.

### `case_implants`

Optional convenience table holding the latest implant info for a case.

Suggested columns:

- `id uuid primary key default gen_random_uuid()`
- `case_id uuid not null unique references cases(id) on delete cascade`
- `source_visit_procedure_id uuid references visit_procedures(id) on delete set null`
- `brand text`
- `model text`
- `diameter_mm numeric(4,1)`
- `length_mm numeric(4,1)`
- `placed_on date`
- `updated_at timestamptz not null default now()`

Why keep this table:

- Implant details can be edited later without scanning all visit history.
- Case detail screens and export become faster and simpler.

### `visit_photos`

Photo metadata. The actual file stays in Supabase Storage.

Suggested columns:

- `id uuid primary key default gen_random_uuid()`
- `owner_user_id uuid not null references auth.users(id)`
- `visit_id uuid not null references visits(id) on delete cascade`
- `case_id uuid not null references cases(id) on delete cascade`
- `visit_procedure_id uuid references visit_procedures(id) on delete set null`
- `bucket_name text not null default 'case-photos'`
- `storage_path text not null unique`
- `file_name text not null`
- `mime_type text`
- `caption text`
- `photo_label text`
- `sort_order integer not null default 1`
- `taken_at timestamptz`
- `created_at timestamptz not null default now()`

Notes:

- `photo_label` can hold simple tags like `pre-op`, `post-op`, `panorama`, `cbct`, or custom stage labels.
- For side-by-side comparison, query by `case_id` or by patient + `tooth_code`, then group by visit/procedure/date.

## Storage design

Create one private bucket:

- `case-photos`

Recommended path pattern:

```text
{owner_user_id}/{patient_id}/{case_id}/{visit_id}/{uuid-filename}.jpg
```

Why:

- Easy ownership checks in storage policies.
- Predictable structure for cleanup when deleting a case or visit.

## RLS strategy

Because you are the only user, the cleanest policy is still owner-based:

- Core tables (`patients`, `cases`, `case_plan_steps`, `visits`, `visit_photos`) store `owner_user_id`.
- Insert policy: `owner_user_id = auth.uid()`
- Select / update / delete policy: `owner_user_id = auth.uid()`

Static seed tables:

- `tooth_positions`
- `treatment_templates`
- `treatment_template_steps`

These can either:

- allow `select` for authenticated users only, or
- disable RLS if you prefer them fully static

## Indexes

Recommended indexes:

- `patients(owner_user_id, created_at desc)`
- `patients using gin (full_name gin_trgm_ops)`
- `cases(patient_id)`
- `cases(tooth_code)`
- `cases(status)`
- `case_plan_steps(case_id, step_order)`
- `case_plan_steps(status, planned_date)`
- `visits(case_id, visited_on desc)`
- `visit_procedures(procedure_type)`
- `visit_photos(case_id, created_at desc)`

## Suggested dashboard queries

### Overdue follow-ups

`case_plan_steps`

- `status = 'pending'`
- `planned_date < current_date`

### Upcoming follow-ups

`case_plan_steps`

- `status = 'pending'`
- `planned_date between current_date and current_date + interval '14 days'`

### Search by patient name

Join `cases -> patients`, filter `patients.full_name`

### Search by procedure

Join `cases -> visits -> visit_procedures`, filter `visit_procedures.procedure_type`

## Export strategy

For CSV export, a flattened query or SQL view is enough.

Recommended export fields:

- patient name
- chart number
- case number
- tooth code
- case status
- template
- next planned date
- overdue days
- latest visit date
- latest procedure
- implant brand / model / diameter / length
- attention alert

## Key implementation assumptions to confirm before coding

1. Tooth numbering uses FDI (`11`, `12`, `21`, `36`, etc.).
2. One visit can contain multiple procedures.
3. We keep a separate `case_implants` table so implant info is easy to edit later.
4. Treatment templates are seeded in the database, not hardcoded only in React.
5. Photo bucket is private; app reads with signed URLs.
